## Context

当前地图系统使用单层 TileMapLayer（Ground10 + Ground11）实现 2D 网格地图。人物位置用 `Vector2i` 表示，寻路在纯 2D 空间进行。

需要扩展为多层地图，支持楼梯上下楼，实现立体效果。

## Goals / Non-Goals

**Goals:**
- 支持任意数量楼层（当前实现 2 层）
- 楼梯放在 obstacle layer，custom_data 标记 `is_stairs`
- 楼梯周围可走格子为出口/入口，无需记录 exit 坐标
- 人物渲染根据楼层应用 Y 偏移（半格 = 8px）
- 点击检测支持多层，按视觉距离选择

**Non-Goals:**
- 不支持同一格子多层重叠（level=N 有地面，其他 level 该位置无地面）
- 不支持楼梯 target_grid 配置（周围自动检测）
- 不处理跨层视线/攻击（只处理移动）

## Decisions

### 1. LevelManager 独立类

**决定**: 创建 `level_manager.gd` 独立类管理多层数据

**理由**: 
- 职责单一，便于测试和复用
- 避免 main.gd 过于臃肿
- 未来扩展层数无需修改核心逻辑

**替代方案**: 直接在 main.gd 用 Dictionary → 耦合度高，难维护

### 2. 楼梯检测逻辑

**决定**: 
- 上楼：当前格 is_stairs → 周围上层可走格子可上
- 下楼：周围下层 is_stairs → 可下

**理由**:
- 楼梯周围自动检测，无需配置 exit 坐标
- 符合"level=N 有地面，其他 level 该位置无地面"的约束

**替代方案**: 楼梯记录 target_grid → 增加配置复杂度，收益不大

### 3. 寻路空间模型

**决定**: 路径节点为 `{grid: Vector2i, level: int}`，BFS/A* 在 (grid, level) 空间搜索

**理由**:
- 统一处理同层移动和跨层移动
- 楼梯算 1 步，代价一致

### 4. 渲染偏移

**决定**: 每层配置 `y_offset`，人物 world_pos.y += offset

**理由**:
- 简单直接，无需坐标系转换
- -32px 偏移 = 2 格高度，产生明显立体感

### 5. 点击检测

**决定**: 遍历所有层，计算 click_grid，选视觉距离（Y 差）最小的层

**理由**:
- 支持跨层点击（楼上点楼下）
- 视觉直觉：点哪个看到哪个

## Risks / Trade-offs

**[风险] 多层寻路性能** → 当前 BFS 复杂度 O(V+E)，层数增加后节点数翻倍。缓解：限制 max_steps，未来可优化为 A*。

**[风险] 楼梯检测边界情况** → 楼梯周围无上层地面时无法上楼。缓解：地图设计时确保楼梯周围有可走格子。

**[权衡] 楼梯不记录 exit** → 简化配置，但楼梯必须周围有出口。可接受，符合设计约束。
